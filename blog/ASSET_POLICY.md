# Asset Policy

블로그 자산은 각 포스트 폴더의 `assets/`에서 관리하고, 빌드 시 최종 `/blog/assets/*` URL로 평탄 복사한다.

핵심 원칙은 단순하다.

- SVG는 장식이 아니라 `정보를 압축한 두 번째 본문`이다.
- 첫 번째 SVG 하나만 봐도 글의 핵심 질문과 판단 흐름이 보여야 한다.
- 같은 템플릿 박스를 30개 글에 반복하지 않는다.

## 위치 규칙

- 자산은 반드시 해당 포스트 폴더 아래 `assets/`에 둔다.
- 루트 `blog/assets/`는 사용하지 않는다.
- 본문 참조는 항상 `./assets/...` 상대경로를 쓴다.

## 파일명 규칙

- 모든 자산 파일명은 포스트 번호 접두사를 붙인다.
  - 예: `012-depreciation-flow-map.svg`
- 다른 포스트와 같은 파일명을 쓰면 빌드가 실패한다.

## SVG 시스템 2.0

글당 기본값은 `5개`다. 다만 5개를 채우는 것보다 역할이 분명해야 한다.

권장 역할은 아래에서 고른다.

- `hero-map`: 글의 질문과 답을 한 화면에 압축
- `decision-tree`: 언제 무엇을 먼저 볼지 분기
- `comparison-matrix`: 좋은 경우 vs 위험한 경우
- `timeline-flow`: 전개 순서, 파이프라인, 보고서 읽기 순서
- `signal-card`: 경고 신호, 자주 놓치는 패턴
- `checklist-board`: 마지막 행동 요약
- `evidence-layer`: 본문에서 말한 근거 레이어를 다시 정리

하나의 글에서 같은 역할 SVG를 2개 이상 반복하지 않는다.

## 캔버스 규칙

`800x430` 하나만 반복하지 않는다. 정보 밀도에 따라 캔버스를 고른다.

- 넓은 흐름도: `1200x720`
- 비교표/매트릭스: `1200x820`
- 체크리스트/카드형: `1080x720`
- 모바일 친화 소형 카드: `960x640`

기존 글은 당장 전면 교체하지 않아도 되지만, 리라이트 시에는 동일 캔버스 반복을 줄인다.

## 정보 밀도 규칙

- 첫 SVG는 제목만 예쁜 그림이 아니라, 최소 1개의 핵심 판단 문장을 포함해야 한다.
- 정보형 SVG는 `실질 라벨 6개 이상`을 기본으로 한다.
- 체크리스트형 SVG는 `질문/항목 5개 이상`이 직접 보여야 한다.
- 텍스트가 3개 이하인 SVG는 장식 자산으로 간주하고 우선 교체 대상에 넣는다.

## 시각 품질 규칙

- 색은 강조용으로만 쓰고, 정보 구분은 위치/레이어/라벨로도 드러내야 한다.
- `좋은 vs 위험`, `충분 vs 부족`, `기본 vs 심화`처럼 대비가 있는 글은 색 대비를 강하게 쓴다.
- 박스만 늘어놓지 않는다. 흐름, 연결선, 강조 배지, 요약 문장을 같이 쓴다.
- 한국어 본문 글이면 SVG도 한국어를 우선한다. 단, `DART`, `EDGAR`, `MD&A`, `10-K` 같은 공식 표기는 유지한다.
- 모바일 360px 폭에서도 첫 줄 질문과 핵심 라벨이 읽혀야 한다.

## 접근성 / XML 규칙

- SVG 안의 `&`, `<`, `>`는 반드시 XML escape를 지킨다.
- 가능하면 `<title>`과 `<desc>`를 넣는다.
- alt 문구는 설명형으로 쓴다.
- `image`, `diagram`, `graphic` 같은 일반어 반복은 피한다.
- XML 파서에서 깨지는 SVG를 남기지 않는다.

## 리라이트 우선 기준

아래 조건에 걸리면 우선 교체 또는 재작성한다.

- XML 파서 오류가 있다.
- 텍스트 노드가 3개 이하라서 정보량이 너무 적다.
- 모든 글에서 같은 박스 템플릿처럼 보인다.
- 첫 SVG가 글의 질문을 설명하지 못한다.
- 본문 내용과 SVG 역할이 겹쳐서 새 정보를 주지 못한다.

## 검증 체크리스트

1. 본문 자산 경로가 `./assets/...`인지 확인한다.
2. 파일명 접두사가 포스트 번호와 일치하는지 확인한다.
3. `uv run python -X utf8 scripts/auditBlog.py`로 XML/밀도 이슈를 확인한다.
4. `npm run build` 후 `landing/build/blog/assets`에 복사됐는지 확인한다.
5. 자산 파일명 충돌이 없는지 확인한다.
6. 모바일에서도 핵심 라벨이 읽히는지 확인한다.
