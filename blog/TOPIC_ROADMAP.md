# Topic Roadmap

이 문서는 블로그를 `글 수 확장`보다 `사람에게 바로 도움이 되는 검색 허브`로 키우기 위한 운영 문서다.

현재 기준은 워크트리 기준 70개 포스트다.

## 지금 상태 요약

강한 축:

- DART / EDGAR 전체 구조
- 사업보고서 읽기 기본 흐름
- 대주주 / 특수관계인 / 임원 보수 / 주주환원 / 주총
- 생산능력 / 건설중인자산 / 감가상각
- 매출채권 / 재고 / 이익의 질
- OpenDART / XBRL / 원문 다운로드 파이프라인
- 영업현금흐름 / 선수금·계약부채 / 신규사업 검증
- 자기주식 / 제3자배정 / 최대주주 변경 / 합병·분할
- 지급보증 / 담보 / 약정 / 적정 의견 아래 위험 신호
- 감자·주식병합 / 내부회계·감사위원회 / 개발비·무형자산 / 최대주주 주식담보
- 리스부채 / 차입 만기 / 지분법손익 / 영업외손익 / 스톡옵션 희석
- 환율 손익 / 파생상품 / 우선주·RCPS / 계속기업 관련 불확실성 / 판관비 구조
- 팩토링·유동화 / 차입 약정·기한이익상실 / 교환사채·EB / 이연법인세
- 공급망금융 / 감사 전 정정·재감사 / 메자닌 보호조항·리픽싱 / 기타포괄손익 누적
- 매각예정자산·중단영업 / 감사보수·비감사보수 / 메자닌 만기연장·조건변경 / 자본잠식·관리종목 신호
- 비적정 감사의견 / 매출 인식 시점 / 유형자산 손상차손 / 리픽싱 이후 실제 전환·오버행

약한 축:

- 입문 허브 첫 SVG의 질문-답변형 차별화
- 새로 추가한 허브 글들의 후속 내부 링크 정교화
- 반복되는 H2 템플릿의 다양화
- 다음 생산 배치를 위한 새 공백 발굴
- 시리즈별 `다음 글로 이동시키는` 연결성의 미세 조정

## 전수 점검 결과

`scripts/auditBlog.py` 기준으로 지금 가장 중요한 수치는 아래다.

- 포스트 수: `70`
- 1,200단어 미만 포스트: `0`
- SVG 4개 미만 포스트: `0`
- FAQ 없는 포스트: `0`
- 체크리스트성 헤딩이 없는 포스트: `0`
- XML 파서 기준 깨진 SVG: `0`
- 텍스트 노드 3개 이하 저밀도 SVG: `0`
- 가장 많이 반복된 H2:
  - `FAQ`
  - `정리`
  - `참고한 공식 자료`
  - `10분 체크리스트`

이 수치가 말하는 핵심은 단순하다.

- `기초 허브 글`, `시리즈 구조`, `SVG 체계`는 기준선에 도달했다.
- 다음 병목은 `빈 시리즈`보다 `새 공백 선정`과 `첫 SVG 설명력 차별화`다.

## 운영 방향 전환

이제부터는 `기존 허브 유지 보강`과 `새 공백 생산`을 같은 우선순위로 둔다.

### 우선순위 1

- 검색형 입문 글의 첫 SVG 질문-답변형 강화
- 신규 허브의 후속 내부 링크 정교화
- 시리즈 첫 글과 끝 글 사이 이동 링크 미세 조정

### 우선순위 2

- 새로 열린 공백을 실전 검색 의도로 채우기
- 공식 자료 중심 질문형 허브 `071` ~ `074` 생산

### 우선순위 3

- 반복되는 H2 템플릿 다양화
- 기존 글과 겹치지 않는 질문만 추가

## 사람 도움 중심 개편 원칙

- 너무 깊게 들어가기보다 `지금 무엇을 봐야 하나`를 먼저 답한다.
- 입문 글은 정의보다 `첫 행동`을 먼저 준다.
- 판단 글은 공식 설명보다 `좋은 경우 vs 위험한 경우`를 먼저 보여준다.
- 파이프라인 글은 기능 나열보다 `무엇을 어떤 순서로 붙여야 하는지`를 먼저 준다.
- 같은 시리즈라도 구조를 똑같이 복제하지 않는다.

## AI 검색 노출 원칙

공식 가이드 기준으로 중요한 것은 새 AI 전용 문법이 아니라 `사람 기준의 유용한 본문`이다.

- 첫 문단에서 질문에 직접 답한다.
- 본문 안에 정의, 판단 기준, 다음 행동이 같이 있어야 한다.
- robots, sitemap, JSON-LD는 계속 유지하되, 그것만으로 해결된다고 보지 않는다.
- OAI-SearchBot, Google-Extended, ClaudeBot, PerplexityBot 허용 상태를 유지한다.
- 본문 링크가 시리즈 구조를 실제로 설명해야 한다.

## 다음 신규 글 우선순위

### 1. 리스 개정과 세일앤리스백은 누구에게 유리한가

- 이유:
  - `047`, `055`, `063` 다음 단계로 부채 경감처럼 보이는 이벤트와 현금흐름 착시를 같이 읽게 만들 수 있다.
  - 리스 재협상과 자산 매각을 별도 뉴스가 아니라 같은 유동성 대응 구조로 묶기 좋다.

### 2. 감사 전 내부결산 오류는 어디서 먼저 드러나나

- 이유:
  - `044`, `060`, `067` 다음 단계로 내부통제, 정정공시, 비적정 의견 사이의 연결을 더 직접적으로 설명할 수 있다.
  - 사용자가 실제로 많이 겪는 `왜 잠정 숫자가 자꾸 바뀌나` 질문을 정면으로 받기 좋다.

### 3. 재고평가손실과 저가수주 압박은 어떻게 이어지나

- 이유:
  - `030`, `054`, `068` 다음 단계로 마진 훼손과 회계 보수화가 한 번에 드러나는 상황을 더 직접적으로 다룰 수 있다.
  - 제조·건설형 회사에서 사람들이 가장 자주 묻는 `매출은 있는데 왜 남는 게 없나` 질문과 잘 맞는다.

### 4. 메자닌 조기상환 요구와 유동성 압박은 어디서 먼저 보이나

- 이유:

  - 조기상환 요구는 공시와 재무 숫자를 동시에 흔드는 실전 질문이라 사용자 효용이 높다.

## 카테고리별 운영 방향

### 01-disclosure-systems

- 입문 허브는 `DART 첫걸음`
- 실전 확장은 `EDGAR raw filing`, `기업 이벤트·자금조달`
- 제도 설명보다 클릭 순서와 후속 확인 흐름을 우선한다

### 02-report-reading

- 첫 글은 `전체 읽기 순서`
- 다음 글은 `섹션별 실전 판단`
- 감사, 우발부채, 리스크, MD&A는 서로 교차 링크한다

### 03-financial-interpretation

- `설비투자와 고정비`와 `재고·채권·이익의 질` 두 축으로 단순화한다
- 숫자 자체보다 숫자 조합과 해석 실수를 강조한다
- 세금, OCI, 운전자본 금융기법처럼 headline을 흐리는 보정 레이어를 우선 발굴한다

### 04-data-automation

- endpoint 소개보다 `수집기 설계`를 쓴다
- corp_code, rcept_no, raw filing, XBRL, notes를 레이어로 나눠 설명한다
