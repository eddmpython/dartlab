# 066 Normal Form 실험

## 목표

마크다운 테이블 수평화의 새로운 패러다임 검증:
모든 테이블을 먼저 **(row_key, col_key, value) 정규형**으로 변환한 후, 정규형끼리 매칭하여 수평화.

## 핵심 아이디어

```
원본 마크다운 테이블
    ↓ toNormalForm()
(row_key, col_key, value) 트리플 리스트
    ↓ header_sig 기반 클러스터링
같은 서브테이블 그룹끼리 모음
    ↓ horizontalize()
항목 × 기간 DataFrame
```

## 003 진단 결과 — 처음 실패한 근본 원인 3가지

1. **서브테이블 분리 부재**: 전체 topic의 모든 테이블을 합산 → 항목 폭발 → 목록형 감지
   - audit: 5개 서브테이블 230항목 → 기존은 blockOrder별 분리
   - 해결: header_sig 기반 클러스터링
2. **복합키 폭발**: matrix의 col_key가 많을 때 row_key × col_key > 50
   - salesOrder: 31항목 × 3 col_key = 93 compound
   - 해결: col_key ≤ 5면 compound, 아니면 pipe join fallback
3. **종속기업 등 대규모 목록**: companyOverview 1195항목
   - 해결: 클러스터 단위로 분리 → 소규모 클러스터만 수평화

## 정규형 패러다임 최종 분석

### 장점

1. **구조 분류 불필요**: multi_year/key_value/matrix 3가지 분류가 하나의 파이프라인으로 통합
   - multi_year: 기수 행 감지 → 당기 추출 → col_key="value" 단일
   - flat: 헤더를 col_key로 자동 사용
2. **항목 매칭 정확도 향상**: (row_key, col_key) 복합키로 동일 항목 식별이 더 정확
   - 기존: "등기이사" 하나의 항목에 인원수/승인금액이 pipe join
   - 정규형: "등기이사_인원수", "등기이사_승인금액" 독립 추적
3. **확장성**: 새로운 테이블 구조가 나타나도 같은 파이프라인으로 처리
4. **89.2% topic-level 성공률**: 기존의 블록 단위 54.7%와 직접 비교 불가하나 방향성 동등+

### 한계

1. **서브테이블 클러스터링이 여전히 필수**: 정규형 자체로는 "어떤 서브테이블끼리 같은 그룹인가"를 해결하지 못함. header_sig 기반 클러스터링이 반드시 필요.
2. **multi_year 당기 추출 로직이 여전히 필요**: "제N기" 패턴 인식, 기수→연도 변환 등 기존과 동일한 도메인 로직이 정규형 안에 그대로 들어감.
3. **이력형/목록형 감지 로직이 동일**: Jaccard 겹침률, 항목 수 50개 임계값 등 기존과 완전히 같은 휴리스틱 필요.
4. **col_key 폭발 대응이 ad-hoc**: col_key ≤ 5면 compound, 아니면 pipe join — 이 임계값이 임의적.

### 핵심 결론

> **정규형 패러다임은 "새로운 패러다임"이 아니라, 기존 로직의 다른 추상화 계층이다.**
>
> 실질적으로 수평화 성공을 결정하는 것은:
> 1. 서브테이블 클러스터링 (header_sig)
> 2. multi_year 당기 추출 (기수 패턴 인식)
> 3. 이력형/목록형 감지 (Jaccard, 항목 수)
>
> 이 3가지는 정규형이든 기존 방식이든 동일하게 필요하다.
> 정규형은 중간 표현(intermediate representation)으로서 코드 구조의 깔끔함을 제공하지만,
> 성공률을 올리는 핵심은 위 3가지 도메인 로직이다.

### 실용적 판단

- **채택하지 않음**: 기존 파이프라인에 비해 구조적 장점은 있으나, 성공률 향상이 없고 동일한 도메인 로직이 필요
- **부분 채택 가능**: (row_key, col_key) 복합키 추적은 matrix 수평화 품질 개선에 활용 가능
- **학습 가치**: 정규형 분해가 "왜" 어려운지를 명확히 보여줌 — 어려움의 본질은 테이블 변환이 아니라 서브테이블 대응과 도메인 패턴 인식
